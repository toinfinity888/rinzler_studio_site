# Phase 0 Research: Hotel Audit Platform

**Date**: 2026-05-09
**Branch**: `001-hotel-audit-platform`

## R1 — Hosting compatibility: Next.js on o2switch

**Decision**: Target o2switch's cPanel **Node.js Selector** (LiteSpeed +
Phusion Passenger), running Next.js in `output: 'standalone'` mode under
Node 20 LTS. Designate Clever Cloud as a contractually-equivalent French
sovereign fallback if any Next.js feature (e.g., middleware on edge runtime,
ISR with on-demand revalidation, Image Optimization with Sharp on a path
not writable by the cPanel user) cannot run.

**Rationale**:
- o2switch's cPanel offers a Node.js Selector that wraps Phusion Passenger
  on LiteSpeed; it can serve a long-running Node app on a custom port and
  reverse-proxy it to a domain. This is compatible with `next start` from a
  standalone build.
- `better-sqlite3` ships native bindings; it builds with `node-gyp` against
  the cPanel-provided Node and does not require root privileges, making it
  viable on shared hosting.
- Plain server actions and route handlers run in the standard Node runtime —
  no edge-runtime dependency.
- French sovereignty is preserved (o2switch hosts in France) and the existing
  marketing site already proves cPanel viability for the studio.

**Risks & mitigations**:
- *Risk:* o2switch's Node app may not allow long-running processes on certain
  shared plans (some hosts kill idle workers). *Mitigation:* keep Next.js
  warm with the platform's keep-alive setting; if unavailable, ping
  `/api/health` from an external scheduler.
- *Risk:* LiteSpeed's request body size cap may limit autosave payloads.
  *Mitigation:* autosave only changed fields, not full submission state.
- *Risk:* No managed cron on shared cPanel for the 36-month auto-purge.
  *Mitigation:* use cPanel's built-in cron tab to `curl` `/api/cron/purge`
  daily with a shared-secret header.

**Alternatives considered**:
- *Vercel EU* — best DX, but Schrems-II perception concerns with French SME
  prospects and inconsistent with the studio's "IA souveraine" pitch (per
  Q2 clarification).
- *Clever Cloud (FR)* — first-class Node + Postgres support, simpler ops than
  shared hosting; held as fallback if o2switch's Node environment proves too
  constrained.
- *Static export + separate API* — defeats the point of Next.js server
  actions; would require a parallel PHP or external Node API.

## R2 — ORM and database: Drizzle + better-sqlite3

**Decision**: Drizzle ORM with `better-sqlite3` as the V1 driver. Schema is
the single source of truth in `db/schema.ts`; migrations generated by
`drizzle-kit`.

**Rationale**:
- SQLite is appropriate for ≤ 100 active projects, single-writer (one admin),
  and write volumes dominated by autosave debounced at 1.5 s per client.
- Drizzle is type-safe, schema-first, has zero runtime overhead, and supports
  swapping to Postgres via `drizzle-orm/node-postgres` with minimal changes
  to query call sites.
- `better-sqlite3` is synchronous, ~5x faster than `node-sqlite3` for the
  small-write pattern of autosave, and uses WAL mode for concurrent reads.

**Risks & mitigations**:
- *Risk:* `better-sqlite3` requires native compilation. *Mitigation:*
  `npm rebuild` step in the o2switch deploy script; verify against
  cPanel-pinned Node version.
- *Risk:* SQLite file backup on shared hosting. *Mitigation:* nightly cPanel
  cron tars `audit/data/audit.sqlite` to `~/backups/` after the WAL checkpoint.

**Alternatives considered**:
- *Prisma* — heavier runtime, larger bundle, more opinionated migration
  flow; chosen against to keep cold-start cost low on Passenger.
- *Postgres on a hosted EU service* — premature; SQLite covers V1 scale;
  migration path documented above.
- *PlanetScale / Supabase* — non-EU control plane, would re-introduce the
  Schrems-II perception we deliberately avoided in Q2.

## R3 — Authentication: Auth.js v5 Credentials + Argon2id

**Decision**: Auth.js v5 (NextAuth) with the **Credentials** provider. Verify
passwords using `@node-rs/argon2` (Argon2id, default parameters tuned to
~50 ms verification time). Sessions are JWT-based, scoped to the
`/admin/*` route segment via middleware. One hardcoded admin row seeded at
first deploy.

**Rationale**:
- Q1 clarification: email + password, Argon2id preferred.
- Auth.js v5 is the canonical Next.js auth library; Credentials provider
  sidesteps any external IdP dependency.
- `@node-rs/argon2` is a Rust-backed N-API binding — fast and ships
  prebuilt binaries for Linux x64, avoiding a `node-gyp` build for the
  hashing primitive specifically.
- JWT sessions avoid an extra session table and survive Passenger worker
  restarts.

**Risks & mitigations**:
- *Risk:* JWT secret leakage. *Mitigation:* `AUTH_SECRET` rotated on
  password reset; old JWTs invalidated by version bump in claims.
- *Risk:* Brute-force on the single login. *Mitigation:* IP-bucket rate
  limiting (10 attempts / 15 min) at the route handler; lockout warning
  after 5 failed attempts.

**Alternatives considered**:
- *Lucia* — leaner, but moving to maintenance mode in late 2025; chose
  Auth.js for ecosystem longevity.
- *Magic-link* — explicitly rejected in Q1 (V1 has no SMTP service).
- *Clerk / WorkOS* — non-EU SaaS, conflicts with sovereignty stance.

## R4 — Form rendering: declarative schema + react-hook-form + Zod

**Decision**: One TypeScript module `lib/form-schema/sections.ts` exports a
typed `Section[]` describing every field, type, validation, and i18n key. A
`SectionRenderer` component iterates the active section's fields and dispatches
to a per-type field component. State is managed by `react-hook-form`; client
validation uses `@hookform/resolvers/zod` against a Zod schema also derived
from the same source.

**Rationale**:
- Satisfies FR-023 and FR-045 (single source of truth, schema-driven, easily
  extensible to new sections / industries).
- Zod schema reused for the JSON export schema (R7) — one definition, three
  consumers (UI validation, server validation, export contract).
- `react-hook-form` keeps re-renders surgical (uncontrolled inputs) — critical
  for the operational-difficulty sliders in Section 3.
- Field-type registry keeps adding new types (e.g., file upload, signature)
  to a single index without touching the renderer.

**Field types supported in V1**: `text`, `email`, `url`, `number`, `select`,
`multiselect`, `slider` (1–10 with labelled poles), `textarea`, `radio-group`
(yes/no/unknown), `system-block` (composite for Section 2's per-system
provider/cost/contract/satisfaction/frustrations).

**Alternatives considered**:
- *Formik* — heavier re-render model, slower with sliders.
- *Building on `<form>` + `useReducer`* — viable but reinvents wheel and
  loses ecosystem (DevTools, error handling).
- *JSON Schema (Ajv) instead of Zod* — Zod chosen for TS-native types and
  smaller bundle.

## R5 — Autosave model

**Decision**: Debounced PATCH-style autosave via a server action
`saveAnswers(token, partial)`. Debounce 1.5 s after the last keystroke;
flush on section navigation and on `beforeunload`. Server action upserts
into `answers` table by `(submission_id, field_id)`. Optimistic UI
indicator with three states: `saving`, `saved`, `offline / will retry`
(localStorage queue).

**Rationale**:
- 1.5 s balances responsiveness against write rate (≤ 1 write per field
  per 1.5 s typing burst).
- Patch payload (only changed fields) keeps autosave under any LiteSpeed
  body cap.
- LocalStorage retry queue covers FR-009 ("offline / will retry") without
  requiring a service worker in V1.

**Concurrency**: Autosave is per-field upsert with a single submission row;
two browser tabs trigger the "updated in another window — reload?" notice
on a stale `submission.updated_at` mismatch (FR edge case).

**Alternatives considered**:
- *Continuous WebSocket* — overkill on shared hosting, adds infra.
- *Save only on section navigation* — fails FR-008's "after a short period
  of inactivity" autosave promise.

## R6 — Access tokens: generation, storage, verification

**Decision**: Generate access tokens with `crypto.randomBytes(24).toString('base64url')`
(192 bits of entropy, well above the FR-002 ≥ 128-bit minimum, 32 URL-safe
chars). Store **only** the SHA-256 hash of the token in the `projects`
table; the plaintext is shown to the admin once at creation and never
persisted. Verification compares hashes in constant time.

**Rationale**:
- Defense in depth: a leaked DB does not leak tokens (they remain
  unguessable hashes, no online PII access).
- Constant-time comparison via `crypto.timingSafeEqual` prevents timing
  attacks.
- 192 bits of entropy ≫ FR-002 minimum.

**Revocation**: Set `projects.token_revoked_at`; verification path returns a
generic "not found" page (FR-006).

**Alternatives considered**:
- *JWT-based access tokens* — would expose project metadata in the URL
  (claims) and complicate revocation; rejected.
- *Plaintext token storage* — leaks on DB compromise; rejected.

## R7 — JSON export schema (versioned)

**Decision**: `lib/export/schema.ts` exports a Zod schema named `ExportV1`
with a top-level `schemaVersion: "audit-export.v1"` discriminator. The export
endpoint serializes `Submission + Project + Notes (excluded by default,
include via query flag) + Scores + Timestamps`. Internal notes are
**excluded by default** from JSON exports to prevent accidental disclosure;
admin opts in with `?include=notes`.

**Rationale**:
- FR-035 mandates a stable, versioned schema.
- Zod-derived schema produces both the runtime validator and the published
  JSON Schema (via `zod-to-json-schema`) for downstream consumers.
- Notes-by-opt-in matches FR-028's strict isolation posture even within the
  admin export surface.

**Future evolution**: `audit-export.v2` may add AI recommendations
(FR-046) as a new top-level optional field; v1 consumers ignore unknown
fields, so this is non-breaking.

## R8 — Scoring heuristics (V1)

**Decision**: Four deterministic heuristics, each producing a 0–100 score
plus a band (`low | medium | high`):

- **Automation Opportunity**: weighted from Section 3 difficulty sliders +
  number of "manual" answers in Sections 3 & 5 + Section 6 interest
  responses.
- **Operational Complexity**: # of distinct systems in Section 2 +
  systems-don't-communicate answer + manual-process count.
- **Modernization Readiness**: Section 6 interest + Section 7 budget /
  timeline / resistance signals (inverse of resistance).
- **Digital Maturity**: # of modern systems in Section 2 + self check-in
  availability + review management process maturity.

Each scorer is a pure function `(answers) => { value: number; band: Band; basis: string[] }`
and lives in `lib/scoring/<name>.ts`. The `basis` array surfaces the answer
ids that drove the score, enabling future explainability in the UI.

**Rationale**:
- Pure functions are trivially unit-testable and deterministic, satisfying
  FR-031.
- Isolated module per FR-033 — replacing a heuristic with an LLM call later
  swaps the file, not the call sites.

## R9 — Animations & motion

**Decision**: CSS-only transitions for V1, no Framer Motion. Use the brand's
`--transition-fast/normal/slow` tokens, view-transitions API for section
swaps where supported (progressive enhancement), `prefers-reduced-motion`
respected globally.

**Rationale**:
- FR-039 caps transitions at 250 ms — easy with CSS, no JS budget cost.
- Keeps client form bundle within the 200 KB budget.
- Framer Motion can be added later for one-off admin micro-interactions
  without architectural change.

## R10 — Brand parity: porting marketing-site tokens to Tailwind

**Decision**: Copy `src/styles/tokens.css` 1:1 into `audit/styles/tokens.css`
(future: turn it into a generated file from a shared `tokens.json` if a
third app appears). In `tailwind.config.ts`, reference the CSS variables in
`theme.extend.colors`, `theme.extend.fontFamily`, `theme.extend.borderRadius`,
`theme.extend.boxShadow`, and `theme.extend.transitionTimingFunction`. This
yields utility classes like `bg-bg-primary`, `text-accent-cyan`,
`shadow-glow-cyan` that resolve to the same HSL/RGB values as the marketing
site.

**Components that mirror the marketing site directly**:
- `Button` ← `.btn` glass-morphism with `backdrop-filter: blur(16px) saturate(160%)`.
- `HelpTooltip` ← `.info-tooltip-trigger` pattern from `calculator.html`.
- `Card` ← `.calc-card` icon-titled pattern (used for each form section header
  and admin project card).
- `Field` ← `.calc-field` label + tooltip-trigger + input layout.
- `GradientText` ← `.calc-gradient` cyan→purple gradient on `<span>`.

**Rationale**:
- Direct mirror means a hotel client landing on the audit form
  immediately recognizes the studio brand from prior touchpoints.
- Tailwind referencing CSS variables (rather than redeclaring values)
  guarantees a single update site if the marketing tokens evolve.

## R11 — Auto-purge scheduling (CNIL 36-month rule)

**Decision**: A small `lib/purge/sweep.ts` runs SQL:
`UPDATE projects SET status='purged', ... WHERE last_admin_activity_at < now()-interval 36 months AND ongoing_engagement = 0`,
deletes related answers / notes / scores, and writes one audit-log row per
purged project. Triggered by:

1. **Primary**: o2switch cPanel cron daily at 03:15 Europe/Paris hitting
   `POST /api/cron/purge` with header `X-Cron-Secret: $CRON_SECRET`.
2. **Backup**: on app startup, opportunistic check; no-op if last sweep
   was within 24 h (recorded in `meta` table).

**Rationale**:
- Shared-hosting-friendly; no need for a long-running scheduler process.
- The endpoint is idempotent and safe to invoke from anywhere with the
  secret.
- On-startup backup ensures the rule is enforced even if cron is misconfigured.

## R12 — i18n posture for V1

**Decision**: Single FR locale shipped in V1. All client-form copy lives in
`lib/form-schema/fr.ts` keyed by field/section id. Admin UI copy lives in
`lib/admin-i18n/fr.ts`. `next-intl` is **not** installed in V1 to keep the
bundle small; we expose a tiny `t(key)` helper that reads from the FR
dictionary directly. Adding `en.ts` later requires no rendering changes,
satisfying constitution Principle III ("i18n-ready") without paying for
the full `next-intl` runtime up front.

**Alternatives considered**:
- *next-intl* — paid for at install time even if only one locale exists;
  defer until a second locale is real.
- *Hardcoded French in JSX* — explicitly forbidden by FR-023 + Principle III.
